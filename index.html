Chapter 1: Truth Tables

1.1 The Language of Logic (Propositional Languages)

In logic, we start with a basic building block called a sentence (or proposition). For our purposes, a sentence is any statement that can be clearly labeled as true or false.

  • Examples of sentences: “Paul is running,” “Laura has red hair,” “It is raining and it is windy,” “If it’s sunny, I will go to the beach.”
  • Examples of non-sentences: “Is it sunny?” or “Paul, come home!” (Questions and commands are neither true nor false, so they don’t count).

Simple vs. Composite Sentences

  • Atomic (Simple) Sentences: These are basic statements that cannot be broken down any further. Example: “Paul runs.”
  • Composite Sentences: These are made by combining atomic sentences together. Example: “It is raining AND it is a windy day.” This contains two atomic sentences connected by the word “and.”

Connectives and Symbols

In natural language, we use words like “and,” “or,” “if…then,” “not,” and “but” to connect simple sentences. In logic, we call these connectives.

To make things easier to write and analyze, we replace words with symbols:

  • We use letters like p,q,rp, q, rp,q,r to represent atomic sentences (e.g., ppp = “It rains”).
  • We use special math-like characters for connectives (e.g., ∧\land∧ = “and”).
  • So, the sentence “It rains and it is a windy day” becomes the formal expression: p∧qp \land qp∧q.

This expression is called a formula. Formulas are essentially a simplified, artificial language used to represent the exact logical meaning of human language.

What Makes a “Legal” Formula?

Just like English has grammatical rules, our logical language (let’s call it L\mathcal{L}L) has rules. The language L\mathcal{L}L is just a collection of letters (p,q,r...p, q, r...p,q,r...). We assume we have as many letters as we need for whatever problem we are trying to solve.

Using these letters, parentheses ()( )(), and connective symbols (∧\land∧ for “and”, ∨\lor∨ for “or”, →\to→ for “if…then”, ¬\neg¬ for “not”), we can write any string of symbols. However, not every string makes sense. For example, )p\neg\neg q is gibberish.

A string of symbols is only considered a valid formula if it follows these strict step-by-step rules:

  1. Rule 1: Any single letter (p,q,rp, q, rp,q,r) is a valid formula. (We call this an atomic formula).
  2. Rule 2: If you already have valid formulas (let’s call them Formula A and Formula B), you can combine them to make new, larger valid formulas by putting them in parentheses and using a connective. So, (A∧B)(A \land B)(A∧B), (A∨B)(A \lor B)(A∨B), (A→B)(A \to B)(A→B), and (¬A)(\neg A)(¬A) are all valid.

If a string of symbols cannot be built using these exact steps, it is not a valid formula.

Helpful Rules for Reading and Writing Formulas

To make formulas easier to read and write, we use placeholders and shortcuts:

  • Metavariables: We use capital letters like A,B,CA, B, CA,B,C as placeholders to talk about formulas in general.
  • Dropping Brackets: Technically, formulas need lots of brackets to avoid confusion. But to avoid clutter, we usually drop the outermost brackets.
  • Order of Operations (Precedence): Just like in math where multiplication comes before addition, logic has an order of operations. The “NOT” symbol (¬\neg¬) is the strongest and attaches to its letter first. “AND” (∧\land∧) and “OR” (∨\lor∨) are next, followed by “IF…THEN” (→\to→), which is the weakest.
    • Example: ¬p→q∨r\neg p \to q \lor r¬p→q∨r actually means ((¬p)→(q∨r))((\neg p) \to (q \lor r))((¬p)→(q∨r)).
  • Shortcuts: Instead of writing (p1∧(p2∧p3))(p_1 \land (p_2 \land p_3))(p1​∧(p2​∧p3​)), we just write p1∧p2∧p3p_1 \land p_2 \land p_3p1​∧p2​∧p3​.
  • The “If and Only If” Shortcut: The symbol ↔\leftrightarrow↔ is a shortcut for a two-way implication. A↔BA \leftrightarrow BA↔B literally means “(A→BA \to BA→B) AND (B→AB \to AB→A)”.

Measuring Formulas

  • Complexity: We can measure how complex a formula is simply by counting how many connectives it has. A single letter ppp has 0 connectives. A formula like ¬p→q\neg p \to q¬p→q has 2 connectives.
  • Subformulas: These are the pieces that make up a larger formula. For example, the subformulas of ¬p→q\neg p \to q¬p→q are ppp, qqq, ¬p\neg p¬p, and the whole thing itself, ¬p→q\neg p \to q¬p→q.

1.2 Truth Functions

Why do we care about formulas? We use them to figure out if an argument (an inference) is logically correct. Let’s look at two arguments:

Inference I

  1. If it rains, I take an umbrella.
  2. I do not take an umbrella.
  3. Conclusion: Therefore, it does not rain.

Inference II

  1. If it rains, I take an umbrella.
  2. It does not rain.
  3. Conclusion: Therefore, I do not take an umbrella.

An argument is only correct if it’s impossible for the premises (the setup) to be true while the conclusion is false. To figure out which of these is correct, we have to define exactly how “True” and “False” work.

Two-Valued Logic

We use a two-valued system: every sentence is either True (T or 1) or False (F or 0). The truth value of a massive, complicated formula depends entirely on the truth values of its smallest atomic pieces.

Here is exactly how each connective calculates Truth and False (these are called Truth Tables):

1. Conjunction (AND / ∧\land∧)
A sentence like “It is raining AND it is windy” (A∧BA \land BA∧B) is only true if both parts are true. If even one part is false, the whole sentence is false.

  • T and T = T
  • T and F = F
  • F and T = F
  • F and F = F

2. Disjunction (OR / ∨\lor∨)
In everyday language, “or” can be confusing. Does “A or B” mean “A or B, but not both” (exclusive) or “A or B, or maybe both” (inclusive)? In logic, we always use the inclusive OR. Therefore, A∨BA \lor BA∨B is true as long as at least one part is true.

  • T or T = T
  • T or F = T
  • F or T = T
  • F or F = F

3. Negation (NOT / ¬\neg¬)
This one is simple: it flips the truth value. If A is true, ¬A\neg A¬A is false.

  • Not T = F
  • Not F = T

4. Implication (IF… THEN / →\to→)
This is the trickiest one. Think of A→BA \to BA→B as a promise. “If A happens, then B will happen.” (A is the premise, B is the conclusion).
The only way to break a promise is if A happens (True), but B does NOT happen (False). If A never happens in the first place, the promise wasn’t broken, so we consider the statement logically True by default.

  • T →\to→ T = T (Promise kept)
  • T →\to→ F = F (Promise broken! The only time implication is false)
  • F →\to→ T = T (Default true)
  • F →\to→ F = T (Default true)

Mathematical Shortcuts for Truth

If we use 1 for True and 0 for False, we can define the connectives using simple math:

  • AND is the minimum of the two values. min⁡(1,0)=0\min(1, 0) = 0min(1,0)=0.
  • OR is the maximum of the two values. max⁡(1,0)=1\max(1, 0) = 1max(1,0)=1.
  • NOT is 111 minus the value. 1−1=01 - 1 = 01−1=0.
  • IF…THEN (A→BA \to BA→B) is max⁡(1−A,B)\max(1 - A, B)max(1−A,B).

Tautologies, Contradictions, and Satisfiability

By evaluating formulas under every possible True/False scenario, we can categorize them:

  • Tautology: A formula that is always true, no matter what. (Example: “It is raining OR it is not raining”).
  • Contradiction: A formula that is always false. (Example: “It is raining AND it is not raining”).
  • Satisfiable: A formula that is true in at least one scenario.

Building a Full Truth Table

To test a formula, we create a table exploring every possible combination of True/False for its letters. If a formula has nnn letters, the table will have 2n2^n2n rows.

  • 2 letters = 4 rows.
  • 3 letters = 8 rows.
  • 4 letters = 16 rows.

Example: Let’s test the formula (p∨q)→¬r(p \lor q) \to \neg r(p∨q)→¬r (which translates to: “If it’s raining or windy, then Marc doesn’t leave the house”).
Because there are 3 letters (p,q,rp, q, rp,q,r), there are 8 rows (23=82^3 = 823=8). You fill out the table by figuring out the inner parentheses first (just like in math), and then working your way out to the final connective. By doing this, we can see that this sentence is satisfiable (it is true in some scenarios), but it is not a tautology (because it is false in others).

(Returning to our umbrella inferences from earlier: If we build truth tables for them, we will find that Inference I is logically correct, but Inference II is flawed because there is a scenario where the premises are true but the conclusion is false).

1.3 Functional Completeness

Now we ask a big, abstract question: Are the logical connectives we have (∧,∨,¬,→\land, \lor, \neg, \to∧,∨,¬,→) enough?

Imagine you invent a brand-new, completely random logical rule. You create a truth table for it, assigning 1s and 0s randomly to the results column. Could we recreate your random, completely made-up truth table using only our standard formulas?

The answer is Yes. Absolutely all of them. This concept is called Functional Completeness.

How to Prove It (Reverse-Engineering a Truth Table)

Let’s prove this by looking at a specific example. Imagine a truth table with two variables (p1,p2p_1, p_2p1​,p2​) where the result is only True (1) when the variables are different (one is 1, the other is 0). This is exactly how “Exclusive OR” (XOR) works.

Here is the table:

  1. p1=1,p2=1  ⟹  Result=0p_1=1, p_2=1 \implies Result = 0p1​=1,p2​=1⟹Result=0
  2. p1=0,p2=1  ⟹  Result=1p_1=0, p_2=1 \implies Result = 1p1​=0,p2​=1⟹Result=1
  3. p1=1,p2=0  ⟹  Result=1p_1=1, p_2=0 \implies Result = 1p1​=1,p2​=0⟹Result=1
  4. p1=0,p2=0  ⟹  Result=0p_1=0, p_2=0 \implies Result = 0p1​=0,p2​=0⟹Result=0

We want to build a formula that perfectly matches this exact table.
The Trick: We only care about the rows where the result is 1.

  • Look at row 2: p1p_1p1​ is false, p2p_2p2​ is true. We can write this as an AND statement: (¬p1∧p2)(\neg p_1 \land p_2)(¬p1​∧p2​). This statement is only true for row 2.
  • Look at row 3: p1p_1p1​ is true, p2p_2p2​ is false. We can write this as: (p1∧¬p2)(p_1 \land \neg p_2)(p1​∧¬p2​). This statement is only true for row 3.

Now, we just link these “true” rows together with an OR (∨\lor∨):
(¬p1∧p2)∨(p1∧¬p2)(\neg p_1 \land p_2) \lor (p_1 \land \neg p_2)(¬p1​∧p2​)∨(p1​∧¬p2​)

We have just successfully built a formula that perfectly matches the custom truth table.

The General Rule: Disjunctive Normal Form (DNF)

This trick works for literally any truth table, no matter how many variables or rows it has. You just:

  1. Find every row where the final result is True (1).
  2. Write an AND statement for each of those rows (using a regular letter if the variable is 1, and a “NOT” letter if the variable is 0).
  3. Connect all those AND statements together with massive ORs.

Formulas built this way have a special name: Disjunctive Normal Form (DNF). It essentially means “a big OR of a bunch of ANDs.”

(Note: If a truth table is completely full of 0s, we conventionally represent it as p∧¬pp \land \neg pp∧¬p. If it is entirely 1s, we use p∨¬pp \lor \neg pp∨¬p).

Conclusion: We Don’t Need New Symbols

Because we can write any truth table in Disjunctive Normal Form, it proves that for any formula you could possibly imagine, there is an equivalent formula (a formula with the exact same logical meaning/truth table) that can be written using only AND, OR, and NOT.

In an nnn-variable logic system, there are exactly 22n2^{2^n}22n uniquely different truth tables possible. Because of Functional Completeness, we know for a fact that our basic toolset is powerful enough to represent every single one of them. We will never need to invent a new logical connective!

Chapter 2: Understanding SAT Problems

2.1 The Big Picture: What is a SAT Problem?

The Boolean Satisfiability Problem (or SAT for short) asks a very simple question: Given a logical formula, is there a way to assign True (1) or False (0) to its variables so that the whole formula evaluates to True?

If such an assignment exists, the formula is satisfiable.

The Challenge:
In theory, you could just build a truth table and check every possible combination line by line. However, if a formula has nnn variables, the truth table has 2n2^n2n rows. If you have hundreds of variables, testing every combination becomes physically impossible for any computer.

Currently, no one knows if a “fast” (polynomial time) algorithm exists to solve SAT problems. This is one of the biggest unsolved mysteries in computer science (the PPP vs. NPNPNP problem). Technically, SAT is NP-complete. This means if you somehow magically guessed the right answer, a computer could verify it very quickly. But actually finding that answer takes exponential time with our current algorithms.

Why do we care?
SAT problems are everywhere in the real world. We use them to ensure computer chips are wired correctly, to check if software has bugs, to create schedules, and to optimize data. Because they are so useful, computer scientists use clever shortcuts—called heuristics—to solve them. Heuristics don’t guarantee a fast answer for every problem, but they work incredibly well for practical, real-world problems.

Two Main Ways to Solve SAT

Algorithms for solving SAT (and automated theorem proving) generally fall into two categories:

  1. Saturation (The “Gathering Clues” Approach):
    This method combines existing logical statements to generate new ones. The goal is to generate so many consequences that if a contradiction exists, it becomes obvious. The main algorithm here is Resolution (created in 1965). The downside? Storing all these new clues takes up massive amounts of computer memory (exponential space).
  2. Optimized Backtracking (The “Smart Guessing” Approach):
    Instead of generating new rules, this method guesses True/False values for variables one by one. If a guess leads to a dead end (a contradiction), it “backtracks” and tries the other option. The goal is to minimize how often you have to backtrack. The undisputed champion of this approach is the DPLL Procedure, named after its creators (Davis, Putnam, Logemann, and Loveland) in the 1960s. Modern SAT solvers are mostly highly engineered versions of DPLL.

2.2 Preprocessing: Normal Forms

Neither Resolution nor DPLL can read a standard logical formula directly. They require the formula to be organized into a specific, flat structure called a Set of Clauses. Therefore, we must translate our formulas first.

The Building Blocks:

  • Atoms: Basic variables (e.g., p,q,rp, q, rp,q,r).
  • Literals: An atom or its negation (e.g., ppp, ¬r\neg r¬r, qqq).
  • Clauses: A group of literals connected only by ORs (∨\lor∨). (e.g., p∨¬r∨qp \lor \neg r \lor qp∨¬r∨q).
  • Empty Clause (□\square□): Represents an immediate contradiction. It is always false (unsatisfiable).

Two Important Definitions:

  • Logically Equivalent: Formula A and Formula B have the exact same truth table.
  • Equisatisfiable: If Formula A can be satisfied, then Formula B can be satisfied, and vice versa. (They don’t necessarily have the same truth table, but they share the same True/False fate).

Step 1: Negation Normal Form (NNF)

A formula is in NNF if it has no implication arrows (→\to→), and all NOTs (¬\neg¬) are pushed completely inward so they only touch individual atoms.

How to convert to NNF:
Apply the following rewrite rules repeatedly until you can’t anymore. (They keep the formula logically equivalent).

  • Remove arrows: C→D⇝¬C∨DC \to D \leadsto \neg C \lor DC→D⇝¬C∨D
  • Cancel double negatives: ¬¬C⇝C\neg\neg C \leadsto C¬¬C⇝C
  • De Morgan’s Law (Distribute NOT over OR): ¬(C∨D)⇝¬C∧¬D\neg(C \lor D) \leadsto \neg C \land \neg D¬(C∨D)⇝¬C∧¬D
  • De Morgan’s Law (Distribute NOT over AND): ¬(C∧D)⇝¬C∨¬D\neg(C \land D) \leadsto \neg C \lor \neg D¬(C∧D)⇝¬C∨¬D

(Shortcut for exercises: ¬(C→D)⇝C∧¬D\neg(C \to D) \leadsto C \land \neg D¬(C→D)⇝C∧¬D)

Example of converting to NNF:
Let’s convert ¬(((p∨q)→r)∧¬(q∨r))\neg(((p \lor q) \to r) \land \neg(q \lor r))¬(((p∨q)→r)∧¬(q∨r))

  1. Change the first AND to ORs using De Morgan’s: ¬((p∨q)→r)∨¬¬(q∨r)\neg((p \lor q) \to r) \lor \neg\neg(q \lor r)¬((p∨q)→r)∨¬¬(q∨r)
  2. Cancel double negative: ¬((p∨q)→r)∨q∨r\neg((p \lor q) \to r) \lor q \lor r¬((p∨q)→r)∨q∨r
  3. Use the shortcut for a negated arrow: ((p∨q)∧¬r)∨q∨r((p \lor q) \land \neg r) \lor q \lor r((p∨q)∧¬r)∨q∨r

Step 2: Conjunctive Normal Form (CNF)

A formula is in CNF if it is an AND (∧\land∧) of ORs (∨\lor∨). In other words, it is a bundle of distinct clauses.

There are two ways to turn an NNF formula into a CNF formula:

Method A: The Non-Structural Method (Good for small, by-hand problems)
This relies on the distributive laws of math (like FOILing in algebra).

  • C∨(D1∧D2)⇝(C∨D1)∧(C∨D2)C \lor (D_1 \land D_2) \leadsto (C \lor D_1) \land (C \lor D_2)C∨(D1​∧D2​)⇝(C∨D1​)∧(C∨D2​)
  • (D1∧D2)∨C⇝(D1∨C)∧(D2∨C)(D_1 \land D_2) \lor C \leadsto (D_1 \lor C) \land (D_2 \lor C)(D1​∧D2​)∨C⇝(D1​∨C)∧(D2​∨C)

Warning: If you apply this to a large formula, it causes an “exponential explosion” in size.

Method B: The Structural Method (Tseitin Transformation - Good for computers)
Instead of distributing, this method replaces chunks of the formula with brand-new placeholder variables.
If you have a problematic chunk like (D1∧D2)(D_1 \land D_2)(D1​∧D2​) inside an OR statement, you invent a new variable aaa, and replace the chunk with aaa. Then, you add rules linking aaa to D1D_1D1​ and D2D_2D2​:
New Formula: A[replace_with_a]∧(¬a∨D1)∧(¬a∨D2)A[replace\_with\_a] \land (\neg a \lor D_1) \land (\neg a \lor D_2)A[replace_with_a]∧(¬a∨D1​)∧(¬a∨D2​)

This keeps the formula size small (linear growth). It changes the exact truth table (because we added a new variable aaa), but the new formula is equisatisfiable to the original, which is all we care about for SAT solving.

2.3 The Resolution Calculus Algorithm

Once we have our formula in CNF, we can treat it as a set of clauses: C0={C1,C2,...Ck}C_0 = \{C_1, C_2, ... C_k\}C0​={C1​,C2​,...Ck​}. We want to know if this set is consistent (satisfiable).

The Resolution Rule:
This algorithm has only one rule. If you have a clause containing ppp, and another clause containing ¬p\neg p¬p, you can cancel ppp out and merge the rest of the clauses together.

  • Rule: (p∨C)(p \lor C)(p∨C) and (¬p∨D)  ⟹  (C∨D)(\neg p \lor D) \implies (C \lor D)(¬p∨D)⟹(C∨D)

When you create a new clause this way, it is a logical consequence of the original two. Adding it to your set doesn’t change whether the puzzle is solvable.

Example of Resolution:
We have: {(p∨q),(¬p∨q),(p∨¬q),(¬p∨¬q)}\{ (p \lor q), (\neg p \lor q), (p \lor \neg q), (\neg p \lor \neg q) \}{(p∨q),(¬p∨q),(p∨¬q),(¬p∨¬q)}

  1. Combine (p∨q)(p \lor q)(p∨q) and (¬p∨q)(\neg p \lor q)(¬p∨q). The ppp and ¬p\neg p¬p cancel. We get (q∨q)(q \lor q)(q∨q), which is just qqq.
  2. Combine (p∨¬q)(p \lor \neg q)(p∨¬q) and (¬p∨¬q)(\neg p \lor \neg q)(¬p∨¬q). The ppps cancel. We get ¬q\neg q¬q.
  3. Now we have a clause qqq and a clause ¬q\neg q¬q. Combine them. They cancel completely.
  4. We are left with the empty clause (□\square□). This proves the original set is unsatisfiable.

Refutational Completeness:
A major mathematical theorem states that Resolution is “refutationally complete.” This means if a set of clauses is unsatisfiable, playing the resolution game will always, eventually result in the empty clause.

(The proof works by “model generation”: It shows that if you do all possible resolutions and never hit the empty clause, you can reliably build a satisfying truth assignment step-by-step without ever getting stuck).

Why don’t we always use Resolution?
Memory. As you generate new clauses, the list gets massively long. However, Resolution is highly efficient for a specific type of formula called Horn Clauses (clauses that have at most one positive literal). If you only have Horn clauses, resolution runs very fast and takes up very little memory.

2.4 The DPLL Procedure

Instead of generating new rules, DPLL is a smart trial-and-error system. It tries to build a satisfying assignment VVV by guessing values and using constraints to force other values.

Terminology:

  • lll means a literal. l‾\overline{l}l means its opposite (if lll is ppp, l‾\overline{l}l is ¬p\neg p¬p).
  • ldl^dld means a literal we randomly decided (guessed) to be true.
  • lpl^plp means a literal we propagated (we were forced to make it true based on previous choices).

The 4 Rules of DPLL:

  1. Decision (Splitting): If you are stuck, pick an unassigned variable and guess that it is True. (Mark it with a ddd, e.g., 1d1^d1d).
  2. Propagation: If your current assignment makes all but one literal in a clause False, the final literal must be True. Force it to be True. (Mark it with a ppp, e.g., 2p2^p2p).
  3. Backtracking: If your current assignment makes an entire clause False (a contradiction), you messed up. Go back to your most recent Decision, undo it, and flip it to the opposite value. This flipped value is now forced (a propagation).
  4. Failure: If you hit a contradiction, but you have no “Decisions” left to undo, the whole formula is UNSATISFIABLE.

(Optional but helpful rule: Pure Literal Rule. If a variable only ever appears as positive in the whole formula, just set it to True. If it only appears as negative, set it to False).

Formula: {(p1∨p2),(p1∨¬p2),(¬p1∨p2),(¬p1∨¬p2)}\{ (p_1 \lor p_2), (p_1 \lor \neg p_2), (\neg p_1 \lor p_2), (\neg p_1 \lor \neg p_2) \}{(p1​∨p2​),(p1​∨¬p2​),(¬p1​∨p2​),(¬p1​∨¬p2​)}
(We will use 1 for p1p_1p1​, 1‾\overline{1}1 for ¬p1\neg p_1¬p1​, etc.)

  1. Start: (Empty)   ⟹  \implies⟹ No forced moves. Guess p1p_1p1​ is True.
  2. State: (1d)   ⟹  \implies⟹ Look at clause 3: (¬p1∨p2)(\neg p_1 \lor p_2)(¬p1​∨p2​). Because p1p_1p1​ is True, ¬p1\neg p_1¬p1​ is False. To satisfy this clause, p2p_2p2​ must be True. Propagate p2p_2p2​.
  3. State: (1d, 2p)   ⟹  \implies⟹ Look at clause 4: (¬p1∨¬p2)(\neg p_1 \lor \neg p_2)(¬p1​∨¬p2​). Our state makes this clause completely False! Contradiction. Backtrack. Undo the decision 1d1^d1d, and force ¬p1\neg p_1¬p1​.
  4. State: (¯1p)   ⟹  \implies⟹ Look at clause 1: (p1∨p2)(p_1 \lor p_2)(p1​∨p2​). Because p1p_1p1​ is False, p2p_2p2​ must be True. Propagate p2p_2p2​.
  5. State: (¯1p, 2p)   ⟹  \implies⟹ Look at clause 2: (p1∨¬p2)(p_1 \lor \neg p_2)(p1​∨¬p2​). Our state makes this completely False! Contradiction.
  6. Result: We have a contradiction, but we have no guessed decisions (ddd) left to backtrack on. Therefore, UNSATISFIABLE.

How to test if a formula is a “Tautology”

A tautology is a formula that is always true no matter what. To test if Formula BBB is a tautology using DPLL, you negate it (¬B\neg B¬B). If DPLL proves that ¬B\neg B¬B is UNSATISFIABLE, it means BBB is a tautology!

2.5 Using SAT-Solvers on Computers

Modern software tools that run advanced versions of DPLL are called SAT-solvers (e.g., MiniSat, Picosat, z3).

The DIMACS Format (Low-level)

To use a classic solver like Picosat, you feed it a text file formatted in the “DIMACS” standard:

  • Comments start with c.
  • The setup line is p cnf [number_of_variables] [number_of_clauses].
  • Variables are represented by positive numbers (1, 2). Negations are negative numbers (-1).
  • Every clause is on a new line and must end with a 0.

Example DIMACS file:
text
c This is an unsatisfiable formula
p cnf 2 4
1 2 0
1 -2 0
-1 2 0
-1 -2 0

The SMT-LIB2 Format (High-level, using z3)

Instead of manually translating everything to CNF clauses, modern solvers like z3 allow you to write formulas naturally using a Lisp-like (prefix) syntax.

  • AND: (and p q)

Chapter 3: Search, Conflict, and Backjumping

In this chapter, we are going to upgrade the standard DPLL algorithm (from the previous chapter) into a much faster and smarter version. We do this by adding advanced techniques used in modern, real-world applications. The most important of these is Backjumping (also known as non-chronological backtracking).

We will present this upgraded procedure in the style of CDCL (Conflict-Driven Clause Learning), which is the standard method used in modern SAT solvers.

3.1 The System State: Search and Conflict

Instead of just looking at formulas, our CDCL algorithm keeps track of a specific “state” as it works. It manipulates a state made of three parts, written as:

(VVV | FFF | α\alphaα)

Here is what each part means in plain English:

  1. VVV (The Notepad / Partial Assignment): This is an ordered list of the variables we have assigned so far (e.g., “A is True, B is False”). Every literal in this list has a label showing why it was assigned:

    • Decided (ldl^dld): We took a guess.
    • Propagated (lpl^plp): We were forced to make this assignment because of the rules (clauses) in our formula.
    • Structure: VVV looks like a timeline. It starts with some forced choices, then a guess (V1,l1dV_1, l_1^dV1​,l1d​), then more forced choices, then another guess (V2,l2dV_2, l_2^dV2​,l2d​), and so on.

  2. FFF (The Rulebook / Set of Clauses): This is the current list of clauses (rules) that must all be satisfied.

  3. α\alphaα (The Status Indicator): This tells us what mode the system is currently in.

    • If α=∗\alpha = *α=∗, we are in Search Status. Everything is fine, and we are looking for the next move.
    • If α\alphaα is a specific clause (e.g., CCC), we are in Conflict Status. This means our current notepad (VVV) violates this specific clause. We call this the conflict clause.

The Setup and The Three Golden Rules (Invariants)

When we start the algorithm, the state looks like this: (∅\emptyset∅ | F0F_0F0​ | ∗*∗)

  • The notepad is empty (∅\emptyset∅).
  • The rulebook contains the original formula (F0F_0F0​).
  • The status is “Search” (∗*∗).

As the algorithm runs and changes the state, it always maintains three unbreakable rules (invariants):

  1. The Core Problem Doesn’t Change: The current rulebook (FFF) is always logically identical to the starting rulebook (F0F_0F0​). We might add new rules, but they are just logical consequences of the original ones.
  2. Conflicts are Real: If we are in Conflict Status (where α=C\alpha = Cα=C), it means two things: clause CCC is a valid logical consequence of our original rules, AND our current notepad (VVV) makes clause CCC completely false.
  3. Every Forced Move has a Reason: If a literal lll was forced (propagated), it means there is a specific clause in our rulebook (like D∨lD \lor lD∨l) where all the other variables (DDD) were already made false by our notepad. Therefore, lll had to be true.

How does it end?

  • Success (SAT): We reach a search state (VVV | FFF | ∗*∗) where all variables are assigned, and no rules are broken.
  • Failure (UNSAT): We hit a conflict, and the rules prove the problem is impossible to solve.

(Note: The rules below can be applied in almost any order, but guessing [the Decide Rule] is always the absolute last resort).

3.2 Search Rules (When α=∗\alpha = *α=∗)

When the status is ∗*∗, we are happily searching. We have three moves available:

  1. Propagate Rule (The Forced Move):

    • Condition: We have a clause D∨lD \lor lD∨l. Our notepad (VVV) makes DDD completely false, and lll hasn’t been assigned yet.
    • Action: We are forced to make lll true. We add lll to our notepad as a propagated literal (lpl^plp).

  2. Conflict Entering Rule (Hitting a Wall):

    • Condition: There is a clause CCC in our rulebook, and our current notepad (VVV) makes the entire clause completely false.
    • Action: We must stop searching. We change our status indicator from ∗*∗ to CCC. We are now in Conflict Status.

  3. Decide Rule (Taking a Guess):

    • Condition: We can’t propagate anything, and we aren’t in a conflict.
    • Action: Pick an unassigned variable lll that exists in our rules, and guess that it’s true (or false). We add it to our notepad as a decided literal (ldl^dld).

3.3 Conflict Rules (When α\alphaα is a clause)

When we hit a conflict, we must figure out what went wrong and fix it. We have three possible moves here:

  1. Failure Rule (Game Over):

    • Condition: The conflict clause (CCC) is completely empty, OR we hit a conflict but our notepad has zero guesses in it (meaning the rules contradicted themselves right out of the gate).
    • Action: The algorithm stops and declares the formula UNSATISFIABLE.

  2. Explain Rule (Finding the Root Cause):

    • Concept: When a conflict happens, the conflict clause contains variables that are currently false. Some of those variables were forced to be false. We want to find out why they were forced, to trace the conflict back to our bad guesses.
    • Condition: We are stuck on a conflict clause (let’s call it l∨Cl \lor Cl∨C). The literal l‾\overline{l}l (the opposite of lll) is on our notepad, and it was a forced (propagated) move. The reason it was forced was because of another clause: D∨l‾D \lor \overline{l}D∨l.
    • Action: We combine the two clauses using a logical step called resolution. We smash together D∨l‾D \lor \overline{l}D∨l and l∨Cl \lor Cl∨C to create a new, combined conflict clause: D∨CD \lor CD∨C. This new clause replaces the old one as our status indicator α\alphaα.
    • Why? We are working backwards through our timeline, unravelling forced moves until we find the actual bad guess that caused the avalanche of forced errors.

  3. Backjumping Rule with Learning (The Time Machine):

    • Condition: By repeatedly using the Explain Rule, we eventually create a new conflict clause (C∨l′C \lor l'C∨l′). Let’s say l′l'l′ is the literal in this clause that we assigned most recently. If we were to jump back in time to right before we made the guess that eventually forced l′l'l′, we could use this new clause to force a completely different, correct path.
    • Action:
      1. Learn: We permanently add this newly discovered conflict clause (C∨l′C \lor l'C∨l′) to our rulebook FFF.
      2. Jump: We clear out our notepad, jumping back in time to an earlier state (V1V_1V1​). We jump back exactly far enough so that l′l'l′ is unassigned, but the rest of the clause CCC is still false.
      3. Propagate: Because we learned the new rule, l′l'l′ is now a forced move. We flip it to its correct value, mark it as propagated, change our status back to Search (∗*∗), and continue.
    • The Magic of Backjumping: Normal backtracking just undoes the very last guess you made. Backjumping realizes that your last guess might have had nothing to do with the conflict. It jumps back straight to the root cause, skipping over (and saving us from exploring) totally irrelevant guesses!

Keeping the Rulebook Clean and Termination

Because we “learn” a new rule every time we backjump, the rulebook FFF will grow. To prevent the computer from running out of memory, real CDCL algorithms delete learned clauses that haven’t been used recently.

Does this algorithm run forever? No.
The algorithm is mathematically guaranteed to finish.

  • In Conflict Mode, the Explain Rule always works backward in time, so it can only run a finite number of times before hitting a guess or failing.
  • In Search Mode, we never get stuck in a loop. Every time we backjump, we learn a new rule that forces us down a path we hadn’t taken before. You can think of the state as a complex odometer; every action (guessing, forcing, or backjumping) clicks the odometer forward, meaning we are constantly making progress toward either a solution or a total failure.

Chapter 4: Semantics for First-Order Languages

4.1 Why Do We Need a New Language? (Motivations)

In previous chapters, we used Propositional Logic to study whether logical inferences were correct. However, propositional logic is quite basic—it only looks at whether whole sentences are true or false. Because of this, there are perfectly logical arguments that propositional logic cannot prove.

Take the classic example:

  1. All men are mortal.
  2. Socrates is a man.
  3. Therefore, Socrates is mortal.

We all intuitively know this is a correct deduction. But if we try to write this using purely propositional logic, we have to assign a single letter to each whole sentence:

  • ppp = “All men are mortal.”
  • qqq = “Socrates is a man.”
  • rrr = “Socrates is mortal.”

The argument becomes: If ppp and qqq are true, then rrr is true (p∧q→rp \land q \rightarrow rp∧q→r).
But in propositional logic, p∧q→rp \land q \rightarrow rp∧q→r is not a tautology (always true). For example, if ppp and qqq are true (1) but rrr is false (0), the statement fails. So, propositional logic tells us this valid argument is invalid!

The Problem: Propositional logic only looks at how whole sentences relate to each other. It cannot look inside the sentences. To prove the Socrates argument, we need a language that understands the internal structure of sentences—words like “all,” “every,” and properties like “is a man.”

The Solution: We need First-Order Logic (FOL). FOL introduces new tools:

  1. Quantifiers: The symbol ∀\forall∀ (Universal Quantifier) means “for all/every.” The symbol ∃\exists∃ (Existential Quantifier) means “there exists at least one.”
  2. Terms: These represent the “objects” we are talking about (e.g., numbers, people, “Socrates”).
  3. Predicates: These represent properties or relationships (e.g., “is a man,” “is less than”).

Using FOL, if P(x)P(x)P(x) means "xxx has property PPP", then:

  • ∀xP(x)\forall x P(x)∀xP(x) means “Every xxx has property PPP.”
  • ∃xP(x)\exists x P(x)∃xP(x) means “There is at least one xxx that has property PPP.”

Let’s rewrite the Socrates example in FOL:

  • Let U(x)U(x)U(x) mean "xxx is a man."
  • Let M(x)M(x)M(x) mean "xxx is mortal."
  • Let sss stand for “Socrates.”

The argument becomes:

  1. ∀x(U(x)→M(x))\forall x (U(x) \rightarrow M(x))∀x(U(x)→M(x)) (For all xxx, if xxx is a man, then xxx is mortal)
  2. U(s)U(s)U(s) (Socrates is a man)
  3. M(s)M(s)M(s) (Socrates is mortal)

With these tools, we can mathematically prove that the conclusion is correct.

4.2 Building the Language (Elementary Languages)

To use First-Order Logic, we must define an artificial language, which we will call an Elementary Language (L\mathcal{L}L). Creating this language is a modeling choice—you build the specific language you need based on the problem you are trying to solve.

An elementary language L\mathcal{L}L is defined by a triple of data (P,F,α)(\mathcal{P}, \mathcal{F}, \alpha)(P,F,α):

  1. P\mathcal{P}P (Predicate Symbols): Used to represent properties or relationships. (e.g., verbs or adjectives like “is a man” or “is greater than”).
  2. F\mathcal{F}F (Function Symbols): Used to represent mathematical operations or data structures (e.g., “plus”, “times”, or “the father of”).
  3. α\alphaα (Arity Function): This simply dictates how many “inputs” or “arguments” a predicate or function takes.
    • A unary predicate (arity 1) takes one input: U(x)U(x)U(x) = "xxx is a man."
    • A binary predicate (arity 2) takes two inputs: L(x,y)L(x, y)L(x,y) = "xxx is less than yyy."
    • Note: A function with an arity of 0 is just a constant (a specific object, like the number 000 or the person “Socrates”). A predicate with an arity of 000 is just a standard propositional letter (like ppp or qqq).

Universal Symbols:
Every first-order language also includes basic tools that never change:

  • Variables: An infinite list of lowercase letters (x0,x1,y0,y1...x_0, x_1, y_0, y_1...x0​,x1​,y0​,y1​...) used as placeholders for objects.
  • Connectives: And (∧\land∧), Or (∨\lor∨), Not (¬\neg¬), Implies (→\rightarrow→).
  • Quantifiers: For all (∀\forall∀), Exists (∃\exists∃).
  • Identity (Optional): If the language includes the equals sign (===), it is called a “language with identity.”

Syntax: Terms vs. Formulas

It is crucial to understand the difference between a “term” and a “formula.”

  • Terms are “Nouns”: They name things in your domain. Variables (xxx), constants (ccc), and functions applied to terms (f(x,c)f(x, c)f(x,c)) are all terms. They do not have a True/False value.
  • Formulas are “Sentences”: They make statements that can be True or False. Applying a predicate to a term makes an atomic formula (e.g., P(x,c)P(x, c)P(x,c)). You can build larger formulas using connectives and quantifiers.

Important Vocabulary

  • Ground (Closed): A term or formula is “ground” if it contains no variables. It refers to specific, concrete things.
  • Free vs. Bound Variables: If a variable xxx is attached to a quantifier like ∀x\forall x∀x or ∃x\exists x∃x, it is bound. If it is just floating without a quantifier, it is free.
  • Sentence: A formula where every variable is bound. There are no free variables.
  • Substitution: We can replace variables with specific terms. We write A(t1/x1)A(t_1 / x_1)A(t1​/x1​) to mean “take formula AAA and replace every free instance of x1x_1x1​ with the term t1t_1t1​.”

4.3 Semantics: Giving Meaning to the Symbols (Structures)

Syntax is just grammar. To know if a formula is actually true, we have to give it a context, or a “world.” In logic, we call this a Structure (A\mathcal{A}A).

A structure A\mathcal{A}A is a pair (A,I)(\mathbf{A}, \mathcal{I})(A,I):

  1. The Domain (A\mathbf{A}A): A non-empty set of “stuff” we are talking about. This could be the set of all integers, the set of all living humans, etc.
  2. The Interpretation Function (I\mathcal{I}I): A dictionary that links our artificial symbols to the actual items in the Domain.
    • It links every constant symbol to a specific object in A\mathbf{A}A.
    • It links every function symbol to an actual math operation.
    • It links every predicate symbol to a specific relationship in A\mathbf{A}A.

Example: Imagine we are doing basic math. The Domain A\mathbf{A}A is the set of natural numbers. We have a constant 000, a function SSS (successor), and an addition function +++.
If we evaluate the term S(0)+S(S(0))S(0) + S(S(0))S(0)+S(S(0)), the interpretation function knows S(x)S(x)S(x) means x+1x+1x+1, and 000 means the number zero. Therefore, the term evaluates to the number 3.

Defining Truth (⊨\models⊨)

Once we have a structure, we can determine if a sentence is true in that structure (written as A⊨A\mathcal{A} \models AA⊨A, read as "Structure A\mathcal{A}A models formula AAA").

The rules are highly intuitive:

  • A⊨P(t1,tn)\mathcal{A} \models P(t_1, t_n)A⊨P(t1​,tn​) is true if the objects represented by those terms actually share that relationship in the real world domain.
  • A⊨A1∧A2\mathcal{A} \models A_1 \land A_2A⊨A1​∧A2​ is true if both parts are true.
  • A⊨¬A1\mathcal{A} \models \neg A_1A⊨¬A1​ is true if A1A_1A1​ is false.
  • Quantifiers:
    • A⊨∀xA1\mathcal{A} \models \forall x A_1A⊨∀xA1​ is true if the formula holds true no matter which object from the domain you plug in for xxx.
    • A⊨∃xA1\mathcal{A} \models \exists x A_1A⊨∃xA1​ is true if you can find at least one object in the domain that makes the formula true.

Satisfiability

  • A formula is satisfiable if there is at least one structure (world) where it is true.
  • A formula is a logical truth (or valid) if it is true in every possible structure.

4.4 Ground Clauses Satisfiability

Before solving complex formulas with quantifiers, let’s look at formulas that have no variables at all (ground clauses).

Because ground clauses don’t have variables, they behave exactly like propositional logic. This means we can “translate” them back into simple letters and use basic SAT solvers (like DPLL/CDCL) to solve them.

Propositional Abstraction:
This is the process of taking a First-Order ground formula and replacing its complex predicates with simple propositional letters (p,q,rp, q, rp,q,r).

  • FOL Ground Clause: ¬P(a)∨P(b)\neg P(a) \lor P(b)¬P(a)∨P(b)
  • Propositional Abstraction: ¬p1∨p2\neg p_1 \lor p_2¬p1​∨p2​ (where p1p_1p1​ stands for P(a)P(a)P(a) and p2p_2p2​ stands for P(b)P(b)P(b)).

The Herbrand Universe:
To test this, we construct a “support domain” out of the language itself, called the Herbrand Universe. It is simply the set of all possible ground terms you can build using the constants and functions in your language. (e.g., c,f(c),f(f(c)),…c, f(c), f(f(c)), \dotsc,f(c),f(f(c)),…).

Crucial Lemma: A set of ground clauses is unsatisfiable in First-Order Logic if and only if its propositional abstraction is unsatisfiable in Propositional Logic.

What this means: If you don’t have variables, you can just turn everything into ppp's and qqq's and let a standard computer algorithm figure out if it’s solvable!

4.5 Herbrand’s Theorem (Dealing with “For All”)

What if our formulas do have variables, specifically Universal Quantifiers (∀x\forall x∀x)?

Our goal is to reduce these universal statements back down to ground clauses so we can use the neat propositional trick from Section 4.4.

To do this, we use the concept of a Substructure. If you have a massive domain (like all real numbers), a substructure is a smaller, self-contained piece of it (like just the integers). Truth generally stays the same when you restrict ground terms to a substructure.

Herbrand’s Theorem:
Let T\mathcal{T}T be a set of universally quantified sentences. T\mathcal{T}T is unsatisfiable if and only if the set of all its possible ground instances is unsatisfiable.

What is a ground instance? It’s what happens when you strip the ∀x\forall x∀x off the front of the formula and replace xxx with every single possible term from the Herbrand Universe.

  • Formula: ∀xP(x)\forall x P(x)∀xP(x)
  • Herbrand Universe: {a,b}\{a, b\}{a,b}
  • Ground Instances: P(a)P(a)P(a) and P(b)P(b)P(b)

Why this is amazing: Herbrand’s theorem says that to prove a universal statement is false, you don’t need to check magical abstract structures. You just need to generate its ground instances, translate them into propositional logic, and run them through a SAT solver.

4.6 The Catch: Infinite Universes (Complements and Examples)

There is a massive difference between Propositional SAT solvers and First-Order SAT solvers.

In Propositional logic, you always have a finite number of combinations to check.
In First-Order logic, if your language has even one function symbol, your Herbrand Universe becomes infinite.

  • If you have constant ccc and function f(x)f(x)f(x), your universe is: {c,f(c),f(f(c)),f(f(f(c))),… }\{c, f(c), f(f(c)), f(f(f(c))), \dots\}{c,f(c),f(f(c)),f(f(f(c))),…}

You cannot pass an infinite number of instances into a SAT solver.

How do we handle this?
We use the Compactness Theorem, which states: An infinite set of sentences is unsatisfiable if and only if a finite subset of it is unsatisfiable.

So, computer algorithms use a Semi-Decision Procedure:

  1. Generate a few ground instances.
  2. Check if they are unsatisfiable.
  3. If yes, stop! You proved it.
  4. If no, generate more ground instances and try again.

The flaw: If the original formula is satisfiable, this process will generate new instances forever and never stop looping. Because of this, First-Order Logic is computationally undecidable (proven by Church’s Theorem).

The EPR Fragment (Effective Propositional Reasoning):
There is a sweet spot. If your language has Predicates and Constants, but NO Function Symbols, the Herbrand Universe is strictly finite. In this specific scenario, First-Order Logic is fully decidable and guaranteed to finish running.

(Note on Identity: === requires special handling. To make equality work mathematically, we have to manually add axioms stating that x=xx=xx=x, that if x=yx=yx=y then y=xy=xy=x, and that if two things are equal, you can swap them inside formulas.)

4.7 Skolemization (Dealing with “There Exists”)

We’ve shown that we can use Herbrand’s Theorem to solve formulas with Universal Quantifiers (∀\forall∀). But what about Existential Quantifiers (∃\exists∃)?

Skolemization is an algorithm that completely eliminates ∃\exists∃ quantifiers from a formula, leaving only ∀\forall∀ quantifiers, without changing whether the formula is satisfiable or not.

Step 1: Prenex Normal Form

First, we must move all quantifiers (∀\forall∀ and ∃\exists∃) to the absolute front of the formula.

  • The string of quantifiers at the front is called the Prefix.
  • The meat of the formula left behind is called the Matrix.

Example:
Formula: ∀xP(x)∧∃yQ(y)\forall x P(x) \land \exists y Q(y)∀xP(x)∧∃yQ(y)
Prenex Normal Form: ∀x∃y(P(x)∧Q(y))\forall x \exists y (P(x) \land Q(y))∀x∃y(P(x)∧Q(y))
(The text provides a list of logical rules for safely pulling quantifiers out to the front).

Step 2: Skolemization

Once the formula is in Prenex Normal Form, we eliminate the ∃\exists∃ quantifiers one by one from left to right.

Rule 1: If ∃y\exists y∃y is at the very front (or not preceded by any ∀\forall∀)
We just replace yyy with a brand new constant (a “Skolem Constant”).

  • Before: ∃y(P(y))\exists y (P(y))∃y(P(y)) (“There exists a person who is the president.”)
  • After: P(c)P(c)P(c) (“Let’s just call this person ‘c’, ‘c’ is the president.”)

Rule 2: If ∃y\exists y∃y comes after a ∀x\forall x∀x
This is trickier. If we say ∀x∃y\forall x \exists y∀x∃y, it means “For every xxx, there is a corresponding yyy.” The value of yyy depends on xxx.
Therefore, we replace yyy with a brand new function (a Skolem Function) that takes xxx as an argument.

  • Before: ∀x∃y(M(y,x))\forall x \exists y (M(y, x))∀x∃y(M(y,x)) (“For every person xxx, there exists a yyy who is their mother.”)
  • After: ∀x(M(f(x),x))\forall x (M(f(x), x))∀x(M(f(x),x)) (“For every person xxx, f(x)f(x)f(x) is their mother.”)

The Final Result

By Skolemizing, we transform ANY First-Order formula into an “equisatisfiable” formula that only uses Universal Quantifiers. We can then turn the Matrix into Conjunctive Normal Form (CNF, or “clausal form”), just like in propositional logic.

Summary Theorem:
For any set of First-Order sentences, there exists an equisatisfiable set of sentences that are entirely universal closures of clauses (formulas with only ∀\forall∀ at the front, and normal CNF clauses inside).

Because of this theorem, we can take any logical problem, Skolemize it to get rid of existentials, generate its ground instances, translate it to propositional logic, and feed it to a SAT solver. This pipeline is the foundation of automated theorem proving!

Chapter 5: Satisfiability Modulo Theories (SMT)

Introduction: Why do we need SMT?

Standard Propositional Logic (Boolean logic using just AND, OR, NOT, and True/False variables) is highly effective, but it has limits. It is often not expressive or concise enough to model real-world problems. For example, trying to encode complex arithmetic—like x+y>10x + y > 10x+y>10—into simple True/False variables requires an enormous amount of effort and computing power.

To solve this, computer scientists developed SMT-solvers (Satisfiability Modulo Theories). Instead of using generic Boolean variables where the meaning is unknown, SMT solvers use “atoms” with precise mathematical or logical meanings (e.g., x=yx = yx=y, or a+b<ca + b < ca+b<c).

Popular SMT-solvers include Z3, Yices, CVC4, and MathSAT. This chapter focuses on how these tools work when evaluating formulas without quantifiers (meaning no “for all” ∀\forall∀ or “there exists” ∃\exists∃ symbols), as this is where the theoretical framework is most stable and efficient.

5.1 The CDCL(T) Procedure

To understand SMT, you first need to understand the concept of a Theory (TTT).
In standard logic, a theory is a set of axioms. But in SMT, we think of a theory semantically: it is an algorithm that knows the “rules” of a specific domain (like math). We say a formula is TTT-satisfiable if it makes sense and evaluates to True within the rules of that specific theory.

For example, x>5∧x<3x > 5 \land x < 3x>5∧x<3 is perfectly fine in standard SAT logic if we just treat them as variables A∧BA \land BA∧B. But in the Theory of Linear Arithmetic, the solver knows this is impossible (unsatisfiable).

Integrating a Theory into CDCL

SMT solvers work by taking a standard CDCL SAT solver (which handles the Boolean logic structure) and attaching a Theory Solver (TTT) (which understands the specific mathematical rules).

We augment the standard CDCL algorithm with three new rules so it can communicate with the Theory Solver:

  1. T-Conflict Rule (Catching math errors):

    • What it does: If the CDCL solver makes a series of variable assignments that work in Boolean logic but break the rules of the theory, the theory solver shouts “Stop!”
    • How it works: If the assigned literals are not TTT-satisfiable, the solver creates a conflict clause. To be efficient, the solver isolates the exact minimal set of assignments that caused the contradiction. This minimal set is called an unsat core.

  2. T-Propagation Rule (Forcing logical conclusions):

    • What it does: The theory solver looks at the current assignments and realizes that math/logic forces another variable to be true.
    • How it works: If assigning AAA and BBB makes CCC mathematically necessary, the solver automatically assigns CCC without the CDCL solver having to guess.

  3. T-Explaining Rule (Explaining the forced conclusions):

    • What it does: If the theory solver forced a propagation (as seen above), and the CDCL solver later hits a conflict and needs to backtrack, the CDCL solver asks the theory solver, “Why did you force that assignment?”
    • How it works: The theory solver provides the mathematical rule (clause) that justified the propagation so the standard conflict-resolution process can continue.

Theory Lemmas: Through this process, the algorithm generates new clauses that are mathematically necessarily true. These are called theory lemmas. If a formula is unsolvable (UNSAT), the solver can prove it by combining the original clauses with these theory lemmas.

Example of CDCL(T) in Action

We are using the Theory of Equality with Uninterpreted Functions (EUF).
Input Formula:

  1. g(a)=cg(a) = cg(a)=c
  2. f(g(a))≠f(c)∨g(a)=df(g(a)) \neq f(c) \lor g(a) = df(g(a))=f(c)∨g(a)=d
  3. c≠d∨g(a)≠dc \neq d \lor g(a) \neq dc=d∨g(a)=d

The CDCL solver treats these as Boolean variables (1, 2, 3, 4). The solver might guess that g(a)=cg(a)=cg(a)=c is True, and f(g(a))≠f(c)f(g(a)) \neq f(c)f(g(a))=f(c) is True.
At this point, the EUF Theory Solver steps in: “Wait! If g(a)=cg(a) = cg(a)=c, then applying function fff to both means f(g(a))f(g(a))f(g(a)) MUST equal f(c)f(c)f(c).”
The Theory solver generates a theory lemma (a conflict clause) to explain this rule to the CDCL solver. The CDCL solver then backtracks, uses this lemma, and tries a different path. Eventually, the solver realizes the whole formula is UNSAT.

5.2 Examples of Decision Procedures (Specific Theories)

Different types of problems require different “Theory Solvers.” Here are the most common ones:

1. Linear Real Arithmetic

  • What it is: Math with real (decimal) numbers, using +++, −-−, ===, and <<<. No multiplication between variables.
  • How it works: It translates into solving systems of linear equations and inequalities. It is highly efficient and runs in polynomial time using classical operations research methods like the Simplex algorithm.

2. Linear Integer Arithmetic & IDL

  • What it is: Same as above, but restricted to whole integers. This is much harder to solve (NP-complete).
  • Integer Difference Logic (IDL): Because full integer math is slow, solvers often use a faster sub-theory called IDL. It only allows equations comparing the difference between two variables against a constant: x−y≤nx - y \leq nx−y≤n.
  • How IDL works:
    1. It rewrites all constraints into the format x−y≤nx - y \leq nx−y≤n.
    2. It builds a directed graph where variables are nodes, and the weights of the edges are nnn.
    3. The formula is satisfiable if and only if there are no negative-weight cycles in the graph. This is quickly solved using the classic Bellman-Ford algorithm.

3. Nonlinear Arithmetic

  • What it is: Introduces multiplication between variables (x×yx \times yx×y).
  • How it works: Highly complex. For integers, it is technically undecidable (due to Gödel’s incompleteness theorems). For real numbers, it is decidable but incredibly slow. Modern SMT solvers only partially support nonlinear math.

4. EUF (Equality with Uninterpreted Functions)

  • What it is: Deals with basic equality and abstract functions where we don’t know what the function actually does, only that if inputs are equal, outputs must be equal (e.g., if x=yx = yx=y, then f(x)=f(y)f(x) = f(y)f(x)=f(y)).
  • How it works (Congruence Closure):
    It uses a highly efficient “union-find” data structure. It puts variables that are equal into the same “bucket” (equivalence class).
    If a=ba=ba=b, they go in the same bucket. If the solver sees f(a)f(a)f(a) and f(b)f(b)f(b), it merges their buckets too. If the formula ever claims that two items in the same bucket are not equal (≠\neq=), the solver outputs UNSAT.

5. Array Theory

  • What it is: Models computer arrays with two main operations: read(array, index) and write(array, index, value).
  • How it works: Array theory is technically undecidable in general, but for finite sets of constraints, it is NP-complete. The SMT solver solves arrays by reducing them to EUF.
    • If it sees a≠ba \neq ba=b, it introduces an index and says read(a, i) != read(b, i).
    • If it sees a write operation, it creates logical clauses ensuring that the newly written array is identical to the old one, except at the specific index written to.

6. Theory of Bitvectors

  • What it is: Models how computer hardware actually handles data (e.g., 32-bit integers). It includes addition, multiplication, bitwise operations (AND, OR, XOR), and combining (concatenating) bits.

5.3 Combination of Decision Procedures (Nelson-Oppen)

The Problem: Real-world code doesn’t stick to one theory. What if you have an array of integers? You need the Array Theory and the Integer Arithmetic Theory to talk to each other.

The Conditions for Mixing: To combine two theories (T1T_1T1​ and T2T_2T2​), two conditions must be met:

  1. Disjoint Signatures: They cannot share any special symbols except the standard equality sign (===).
  2. Stably Infinite: Both theories must be valid even if the universe of possible answers is infinitely large. (Most theories, like math and EUF, are stably infinite).

The Nelson-Oppen Procedure

If those conditions are met, we can combine the theories using a framework called Nelson-Oppen. Here is how it works:

Step 1: Flattening (Purification)
The solver breaks complex, mixed formulas into simple, “flat” pieces that belong strictly to only one theory.

  • Example: If we have f(x+1)=yf(x + 1) = yf(x+1)=y, we have EUF (fff) and Math (+1+ 1+1) mixed together.
  • Flattening: We introduce a new dummy variable ccc. The formula becomes c=x+1c = x + 1c=x+1 (sent to the Math solver) and f(c)=yf(c) = yf(c)=y (sent to the EUF solver).

Step 2: Information Sharing
Now, the two separate theory solvers work on their own pieces of the problem. They are only allowed to communicate by sharing one thing: Equalities between shared variables.

  • If the Math solver figures out that c1=c2c_1 = c_2c1​=c2​, it passes that information to the EUF solver.
  • If the EUF solver figures out that c1≠c2c_1 \neq c_2c1​=c2​, it passes that information to the Math solver.
  • If they both agree the formula is satisfiable and no more equalities can be shared, the whole mixed formula is satisfiable.

Convex vs. Non-Convex Theories

To optimize CDCL with multiple theories, we have to consider whether a theory is convex.

  • Convex Theory: If the theory proves that a variable equals a list of possibilities (x=y1∨x=y2x = y_1 \lor x = y_2x=y1​∨x=y2​), it must be able to prove exactly which one it is. (EUF and Real Math are convex).
  • Non-Convex Theory: The theory can know xxx equals one of two things, but can’t prove which one. (Integer Arithmetic is non-convex; e.g., if 1≤x≤21 \leq x \leq 21≤x≤2, xxx must be 111 or 222, but the math alone doesn’t force a specific choice).

If all theories being combined are convex, the solver saves a massive amount of time because it only has to make decisions about variables that actually appear in the formula. If a theory is not convex (like integers), the solver sometimes has to generate new “guesses” (splitting cases) to find a solution, which slows the process down.

Putting it all together (Example 32 Summary)

When combining theories inside CDCL, the algorithm uses standard SAT guessing, but at every step, both theory solvers check the math/logic. If either solver finds a conflict, it generates a theory lemma. If either solver discovers an equality between shared variables, it propagates that equality to the other solver immediately. This cooperative, synchronized checking allows modern tools to verify highly complex software and hardware efficiently.

When you want to solve a logic or math problem using an SMT (Satisfiability Modulo Theories) solver, you need a way to communicate with it. The standard language for this is called SMT-LIB2. It is a bit more complex than the formats used for older, simpler SAT solvers, but it is incredibly powerful.

To use it, you write your problem in a plain text file (usually with a .txt or .smt2 extension). Each line that starts with a semicolon ; is a comment.

Here is how you define the basic building blocks of a problem:

Types (Sorts)

The solver comes with built-in types like Int (integers), Real (decimals), and Bool (true/false). But you can also create your own custom types, known as “uninterpreted sorts.”

  • Syntax: (declare-sort S 0)
  • What this means: You are telling the solver, “Create a new category of things and call it S.” (The 0 just means it takes no parameters).

Constants (Variables)

You can declare standalone variables (which the solver treats as constants whose exact values are yet to be found).

  • Syntax: (declare-const x Int)
  • What this means: “Create an integer variable named x.”

Functions

You can use built-in math functions (+, -, *), but you can also create your own custom functions.

  • Syntax: (declare-fun f (S Int) Int)
  • What this means: “Create a function named f. It takes two inputs (one of type S and one Int) and spits out an Int.”
  • Note: Predicates (things that check if something is true) are just functions that output a Bool. Built-in predicates include =, <, >, <=, and >=.

Assertions are the heart of the problem. You use them to tell the solver what rules must be true. However, SMT-LIB2 uses LISP notation (Prefix notation). This means the operator goes before the arguments.

  • Instead of f(a, x) < 1, you write: (< (f a x) 1)
  • Syntax: (assert (< (f a x) 1))

You combine rules using and, or, not, and => (implies). You can pass multiple arguments to and and or at once.
You can also use quantifiers:

  • Exists: (exists ((x Int) (y Int)) ) — “There is at least one x and y where this formula is true.”

  • Warning: SMT solvers struggle with quantifiers. Asking a solver to check “every number in infinity” might cause it to run forever or return unknown. Use them carefully!

Giving Commands to the Solver

Once you’ve built your problem, you tell the solver what to do:

  • (check-sat): Tells the solver to figure out if the problem is solvable (Satisfiable or “sat”).

6.2 Transition Systems and Bounded Model Checking

So far, we looked at static math problems. But real software changes over time. To model software, we use a Transition System, which is basically a mathematical board game. It consists of:

  1. State Variables (xxx): The current status of the program (the pieces on the board).
  2. Initial State (ι\iotaι): The starting values of the variables (the starting setup).
  3. Transition Formula (τ\tauτ): The rules linking the current state (xxx) to the next state (x′x'x′).

A state is reachable if you can start at the initial state and take valid steps to get there.

We usually try to solve two types of problems:

  1. Planning: “How do I reach a winning state?” (e.g., getting a wolf, sheep, and cabbage across a river safely).
  2. Safety: “Can I mathematically guarantee the system will never reach a bad/buggy state?”

Bounded Model Checking (BMC)

Because software can loop forever, SMT solvers can’t simulate “infinity.” Instead, we use Bounded Model Checking. We ask the solver: “Can you break the rules in 1 step? How about 2 steps? How about N steps?”

The formula looks like this:
InitialState(x0)∧Step(x0,x1)⋯∧Step(xN−1,xN)∧BadState(xN)InitialState(x^0) \land Step(x^0, x^1) \dots \land Step(x^{N-1}, x^N) \land BadState(x^N)InitialState(x0)∧Step(x0,x1)⋯∧Step(xN−1,xN)∧BadState(xN)

If the solver finds a solution, you found a bug (or the winning moves to your puzzle). If it says unsat, the system is safe up to N steps.

6.2.1 Proving Code is Safe (Pre- and Post-Conditions)

In software verification, we analyze code using Hoare Triples:
{P} Code {Q}

  • P (Pre-condition): What must be true before the code runs.
  • Code (π\piπ): The instructions executed.
  • Q (Post-condition): What must be true after the code finishes.

The goal is to prove that if P is true, and the Code runs, Q will always be true. To do this with an SMT solver, we ask the opposite: "Is it possible for P to be true, the code to run, but Q is FALSE?"
Formula: P∧Code∧¬QP \land Code \land \neg QP∧Code∧¬Q.
If the solver says unsat, your code is safe.

Translating Code to Logic

To turn code into a math formula, we introduce a variable called pc (Program Counter) to track what line of code we are on.

  • Assignments: Code like x = t becomes: “Line number advances, x gets updated to t, and no other variables change.”
  • If/Else: Code like if x > t branches the logic: “(Line number advances AND x > t) OR (Code jumps to the ‘else’ block AND x <= t).”
  • While Loops: Modeled similarly to If statements, but the program counter loops back to the start of the block.

6.2.2 Code Annotations and Verification Conditions

Bounded Model Checking only checks code up to a specific number of loop iterations. To prove code is always safe, no matter how many times a loop runs, we have to break the code into Basic Paths (straight paths through the code from the start of a loop to the end of a loop).

We add Annotations (or Loop Invariants) at the start and end of these paths (called cutpoints).
By creating a formula for a basic path—saying “If we start with this annotation, and run this path, the ending annotation must hold true”—we generate a Verification Condition (VC).

If we can prove that all VCs are unsat (meaning the rules can never be broken), the code is guaranteed to be 100% correct. Finding the right loop annotations is hard, and often requires human help or advanced auto-guessing heuristics, but it is the gold standard of software safety.

Chapter 7: Quantifiers in Arithmetical Contexts

When we want to build algorithms that can automatically solve math problems with constraints (like x+y>5x + y > 5x+y>5), we usually rely on operations research techniques. However, formal logic gives us another fascinating tool: Quantifier Elimination (QE).

While QE isn’t always the fastest method (it can require a lot of computing power), it is incredibly flexible. It allows us to take complex statements filled with variables and quantifiers—like "for all xxx" (∀\forall∀) or "there exists an xxx" (∃\exists∃)—and simplify them.

7.1 General Considerations

What is Quantifier Elimination?
Quantifier elimination is a powerful mathematical trick to prove whether a theory is decidable (meaning an algorithm can always find a true/false answer). The core idea is simple: if you can systematically remove all the variables and quantifiers from a formula, you are left with a “ground formula” (a formula with only actual numbers, like 3<53 < 53<5). Deciding if a ground formula is true or false is trivial.

Formal Definition:
A mathematical theory TTT allows for quantifier elimination if, for every complex formula ψ(x)\psi(x)ψ(x) that contains quantifiers, you can translate it into a completely equivalent formula ψ′(x)\psi'(x)ψ′(x) that has no quantifiers.

If we have an effective way to do this translation, we can test the truth of any arbitrary formula.

The Reduction Test (A Shortcut)

Do we have to figure out how to eliminate quantifiers from every complex formula at once? No. The Reduction Test proves that if we can solve a specific, simple case, we can solve them all.

The Rule: If you can figure out how to eliminate a single “exists” quantifier (∃\exists∃) from a simple list of basic facts joined by “AND” (e.g., ∃y(L0∧L1∧⋯∧Ln)\exists y (L_0 \wedge L_1 \wedge \dots \wedge L_n)∃y(L0​∧L1​∧⋯∧Ln​)), then your theory allows for full quantifier elimination.

Why this works (The Proof Concept):

  1. We can flip “for all” (∀\forall∀) statements into “there exists” (∃\exists∃) statements using basic logic (e.g., ∀x\forall x∀x is the same as ¬∃x¬\neg \exists x \neg¬∃x¬).
  2. We can take any complex formula and rearrange it into a “Disjunctive Normal Form” (DNF)—which is basically a giant list of “OR” conditions, where each condition is a bunch of “ANDs”.
  3. We can then distribute the ∃\exists∃ quantifier across the “ORs”.
  4. This breaks the massive problem down into bite-sized pieces. If we can solve the bite-sized pieces, we can solve the whole thing.

Note: While the DNF method proves QE is possible, converting formulas to DNF causes them to blow up exponentially in size. Therefore, in practice, we try to design algorithms that avoid doing this.

7.2 Real Linear Arithmetic (Working with Real Numbers)

Let’s look at numbers on a continuous number line (Real numbers: R\mathbb{R}R). Our logical language includes 0,1,+,−,<,0, 1, +, -, <,0,1,+,−,<, and ===. (Notice there is no multiplication of variables, so we can only do linear math like 2x+3y2x + 3y2x+3y, not x⋅yx \cdot yx⋅y).

We can prove that Real Linear Arithmetic allows for quantifier elimination using an algorithm called Fourier-Motzkin.

The Fourier-Motzkin Algorithm

Before we run the algorithm, we clean up the math:

  1. We pretend we can divide by integers. Even though division isn’t strictly in our language, we can fake it (e.g., instead of dealing with 3x<y3x < y3x<y, we rewrite it as x<y/3x < y/3x<y/3). This means we treat our formulas as linear polynomials with rational (fraction) coefficients.
  2. We get rid of ≤\leq≤, ≥\geq≥, and ≠\neq= so that our basic formulas only look like: u=0u = 0u=0, 0<u0 < u0<u, or u<0u < 0u<0.

The Goal: Eliminate ∃x\exists x∃x from a group of inequalities.

The Steps:

  1. Clean up negative logic: Change things like “not less than” ¬(0<u)\neg(0 < u)¬(0<u) into “equal to or greater than” (0=u∨u<00 = u \vee u < 00=u∨u<0).
  2. Remove useless variables: If an equation doesn’t contain xxx, pull it outside the ∃x\exists x∃x scope.
  3. Substitute exact matches: If you have an exact equation like 0=kx+p(y)0 = kx + p(y)0=kx+p(y), you can solve for xxx exactly (x=−p(y)/kx = -p(y)/kx=−p(y)/k). You can then delete the ∃x\exists x∃x quantifier entirely and just plug that exact value into the rest of the formula!

The Core Squeeze Logic:
If you do all the above, you are left only with strict inequalities containing xxx. You can organize these into two groups:

  • Lower bounds: Expressions that are smaller than xxx (let’s call them ui<xu_i < xui​<x).
  • Upper bounds: Expressions that are bigger than xxx (let’s call them x<vjx < v_jx<vj​).

The formula is basically saying: “There exists an xxx that is bigger than all the uuu's and smaller than all the vvv's.”

Because real numbers are dense (there is always a real number between any two other distinct real numbers), this statement is true if and only if every lower bound is strictly less than every upper bound.

So, we completely delete xxx and the quantifier, and replace it with:
ui<vju_i < v_jui​<vj​ (for every possible pairing of lower and upper bounds).

Complexity Note: If you have 5 lower bounds and 5 upper bounds, eliminating one xxx creates 25 new equations. Doing this multiple times causes a massive, double-exponential explosion in formula size.

The Testing Points Method (A Faster Approach)

Instead of pairing up every lower and upper bound (which creates huge formulas), we can just “test” specific points.

If we want to know if an xxx exists between bounds, we don’t need to write out every u<vu < vu<v. We can literally just plug a hypothetical xxx into the formula. What xxx do we choose?
We choose the midpoint between the upper and lower bounds: x=(u+v)/2x = (u + v) / 2x=(u+v)/2.

To cover all our bases (including edges and infinity), the set of “test points” we plug in for xxx will be:

  • Averages of bounds: (p+q)/2(p+q)/2(p+q)/2
  • Offsets of bounds: p+1p+1p+1 and p−1p-1p−1
  • The bounds themselves: ppp

By substituting these test points into the formula in place of xxx, we completely remove the variable xxx and the quantifier ∃x\exists x∃x without generating nearly as much mathematical bloat.

7.3 Integer Linear Arithmetic (Working with Whole Numbers)

Now we move from Real numbers (continuous) to Integers (Z\mathbb{Z}Z, discrete numbers). This is much harder.

Why? Consider the formula: ∃x(y=x+x)\exists x (y = x + x)∃x(y=x+x). In plain English, this means "yyy is an even number."
In Real numbers, this is trivial (xxx is just y/2y/2y/2). But in Integers, if yyy is 3, there is no whole number xxx that works. You cannot express "yyy is even" using only addition and subtraction without using a quantifier.

The Fix: To achieve quantifier elimination for integers, we must add a new mathematical concept to our language: Divisibility. We introduce an infinite set of predicates called Dk(x)D_k(x)Dk​(x), which simply means "xxx is exactly divisible by kkk."

By adding divisibility, we create an “extended signature.” The famous algorithm to solve this is Cooper’s Algorithm (1972).

Cooper’s Algorithm Pre-Processing

Before we eliminate xxx, we must massage the formula into a very specific shape:

  1. Eliminate all negations (¬\neg¬) by flipping inequalities (e.g., ¬(s<t)\neg(s < t)¬(s<t) becomes t≤st \leq st≤s).
  2. Change equalities (s=ts = ts=t) into boundaries (s<t+1∧t<s+1s < t+1 \wedge t < s+1s<t+1∧t<s+1).
  3. Shift everything so 000 is on the left (s<ts < ts<t becomes 0<t−s0 < t - s0<t−s).
  4. Consolidate terms into linear polynomials.

The “LCM” Trick:
We need the coefficient of xxx (the number attached to xxx, like the 333 in 3x3x3x) to be exactly 111 or −1-1−1 everywhere it appears.
To do this, we find the Least Common Multiple (LCM) of all the coefficients of xxx in the formula (let’s call it lll). We multiply terms so that xxx always has the coefficient lll. Finally, we replace lxlxlx with just a plain xxx, but we add a strict condition: Dl(x)D_l(x)Dl​(x) (meaning, “this new xxx must be divisible by our LCM”).

The Core Logic of Cooper’s Algorithm

After pre-processing, our formula ψ\psiψ consists of basic building blocks:
x<p(y)x < p(y)x<p(y), \quad p(y)<xp(y) < xp(y)<x, \quad Dk(x+p(y))D_k(x + p(y))Dk​(x+p(y)), \quad ¬Dk(x+p(y))\neg D_k(x + p(y))¬Dk​(x+p(y))

Because we are working with discrete integers, if there is a valid integer xxx that makes the formula true, one of two things must be happening:

Case 1: True for infinitely small numbers (The Left Infinity)
Maybe the formula is true because xxx can be an infinitely large negative number.
We create a hypothetical version of our formula called ψ−∞\psi_{-\infty}ψ−∞​. In this version, we replace any rule that says x<px < px<p with “True” (because a massive negative number is always smaller than ppp), and replace p<xp < xp<x with “False”.

Because divisibility rules (DkD_kDk​) repeat in a cycle, we don’t have to check infinity. We just find the LCM of all the divisibility rules (let’s call it mmm). If the formula works for any infinitely small number, it will mathematically repeat and work for at least one number between 111 and mmm.
So, Case 1 is solved by testing if the formula is true for x=1,2,3,…,mx = 1, 2, 3, \dots, mx=1,2,3,…,m.

Case 2: The “Hitting a Wall” Boundary
If xxx can’t be infinitely negative, then there must be a smallest possible valid integer for xxx.
Where would this integer be? It must sit exactly next to one of the lower-bound constraints in the formula! Let’s call the set of all lower-bound polynomials the BBB-set. The smallest valid xxx must be one of those boundary points bbb, plus a tiny integer step jjj.

The Final Cooper Theorem

We can eliminate the ∃x\exists x∃x quantifier by combining Case 1 and Case 2. The formula ∃xψ(x,y)\exists x \psi(x,y)∃xψ(x,y) is completely logically equivalent to:

Check the repeating infinity cycle:
(Is the infinity-version of the formula true for any integer jjj from 111 to mmm?)
OR
Check the boundaries:
(Is the standard formula true if we plug in b+jb + jb+j for xxx, where bbb is a boundary limit, and jjj is a step from 111 to mmm?)

A.3 Quotient Sets (Grouping by Similarity)

Sometimes in logic and math, we have many different names for the exact same object (synonyms). We need a way to group all these synonyms together into one single “concept.” We do this by creating a Quotient Set.

Equivalence Classes (The “Buckets”)
Imagine a room full of people (Set XXX), and our rule (∼\sim∼) is “speaks the same everyday language.”
This rule naturally divides the room into distinct, non-overlapping groups. All Italian speakers form one group, all French speakers form another.

These groups are called equivalence classes. The class is represented by placing brackets around any person inside it. So, if Alice speaks Italian, [Alice][Alice][Alice] represents the entire group of Italian speakers.

The Quotient Set
The Quotient Set (written as X/∼X / \simX/∼) is simply the set of these new buckets. Instead of a set of people, X/∼X / \simX/∼ is a set of nationalities/languages.

A Mathematical Example: Modular Arithmetic (Clock Math)
SMT solvers (automated logic programs) use this heavily. Let’s group integers based on the remainder they leave when divided by a number nnn.
We write x≡nyx \equiv_n yx≡n​y to mean "xxx and yyy give the same remainder when divided by nnn."

Let’s use n=3n = 3n=3. If we divide numbers by 3, the only possible remainders are 0, 1, or 2. This creates exactly three “buckets” (equivalence classes) for all integers in existence:

  • [0][0][0] bucket: {…,−6,−3,0,3,6,… }\{\dots, -6, -3, 0, 3, 6, \dots\}{…,−6,−3,0,3,6,…} (Divides perfectly by 3)
  • [1][1][1] bucket: {…,−5,−2,1,4,7,… }\{\dots, -5, -2, 1, 4, 7, \dots\}{…,−5,−2,1,4,7,…} (Leaves a remainder of 1)
  • [2][2][2] bucket: {…,−4,−1,2,5,8,… }\{\dots, -4, -1, 2, 5, 8, \dots\}{…,−4,−1,2,5,8,…} (Leaves a remainder of 2)

So, the quotient set Z/≡3\mathbb{Z}/\equiv_3Z/≡3​ contains only three items: {[0],[1],[2]}\{ [0], [1], [2] \}{[0],[1],[2]}.

Doing Math with Buckets
Can we do addition and multiplication on these buckets? Yes, but only if the result is the same regardless of which number inside the bucket we choose to represent it.

Let’s test addition on our modulo-3 buckets: [1]+[2][1] + [2][1]+[2].

  • If we pick 111 from the first bucket and 222 from the second bucket: 1+2=31 + 2 = 31+2=3. The number 3 lives in the [0][0][0] bucket.
  • What if we pick different representatives? Let’s pick 444 from the first bucket and 555 from the second: 4+5=94 + 5 = 94+5=9. The number 9 also lives in the [0][0][0] bucket.
  • Let’s try negative numbers: −2-2−2 from the first bucket and −1-1−1 from the second: −2+(−1)=−3-2 + (-1) = -3−2+(−1)=−3. The number −3-3−3 also lives in the [0][0][0] bucket.

Because the answer always lands in the exact same bucket no matter which representatives we use, we can safely say that for the quotient set itself, [1]+[2]=[0][1] + [2] = [0][1]+[2]=[0].

Proving that operations are “independent of the representatives” is a crucial step in advanced logic proofs, which you will see later in sections 4.6 and 5.2.
